# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A [tree-sitter](https://tree-sitter.github.io/) grammar for [navi](https://github.com/denisidoro/navi) `.cheat` cheatsheet files. The entire grammar lives in a single file, `grammar.js`; everything under `src/` and `bindings/` is generated or boilerplate.

## Commands

- `tree-sitter generate` (or `npm run generate`) — regenerate `src/parser.c`, `src/grammar.json`, `src/node-types.json` from `grammar.js`. **Run this after every `grammar.js` edit** — the bindings compile from the generated C, not from the JS.
- `tree-sitter test` (or `npm test`, `make test`) — run corpus tests.
- `tree-sitter parse <file.cheat>` — parse a file and print the syntax tree; the fastest way to check a grammar change. `example.cheat` is a ready-made sample.
- `tree-sitter highlight <file.cheat>` — exercise the highlight queries.
- `make` — build C static/shared libs (`libtree-sitter-navi.a` / `.dylib`); `make install` installs them.

## Grammar model (`grammar.js`)

navi's format is **line-oriented**: the first character of each line (the "marker") decides the line type. The grammar mirrors this exactly — `source_file` is `repeat($._line)`, and `_line` is a `choice` over the line types followed by a newline.

Marker → rule:
- `%` → `tags` (space-separated `tag` tokens)
- `;` → `comment`
- `#` → `description` (snippet header)
- `@` → `authorship` (metadata)
- `$` → `variable` — `$ name: command --- options`
- anything else → `command` (a runnable line, may embed `<var_ref>` placeholders)

Key tokenization decisions that are easy to break:
- **Precedence disambiguates markers from commands.** All marker lines have `prec(2)`, `command` has `prec(1)`, and `_blank` is the fallback. The `command_text` token's regex explicitly excludes the marker chars (`% ; # $ @ <`) *as the first character* but allows them mid-line.
- **`var_command` is chunked**, not a single regex, so it can stop at the ` --- ` options separator: it matches runs of non-dash/non-newline text plus single dashes at lower precedence.
- **`marker` and `:` are aliased/named** so highlight queries can target them.

When changing line parsing, re-run `tree-sitter generate` and verify with `tree-sitter parse example.cheat` — precedence bugs show up as `ERROR` nodes or lines parsed as the wrong type.

## Highlight queries

`queries/highlights.scm` holds the syntax-highlighting rules. This path matches what `package.json` and `Cargo.toml` reference, so `tree-sitter highlight` and editor integrations pick it up. It maps grammar nodes (`tag`, `comment`, `description`, `var_name`, `placeholder`, etc.) to highlight capture names.

## Tests

Corpus tests live in `test/corpus/*.txt`. Each block is `name` / source / `---` / expected S-expression tree (no positions). Run with `tree-sitter test`. Regenerate the parser before testing if you changed `grammar.js`. To add a case, parse a sample with `tree-sitter parse <file>` and copy the tree (minus the `[row, col]` ranges) into a new block.

## Generated / boilerplate — do not hand-edit

- `src/` — generated by `tree-sitter generate` and **committed** so consumers can build without regenerating. Always re-run `tree-sitter generate` after editing `grammar.js` so the committed `src/` stays in sync.
- `bindings/` — per-language bindings (node, rust, python, go, swift, c) plus `binding.gyp`, `Package.swift`, `Cargo.toml`, `setup.py`, `pyproject.toml`. These are standard tree-sitter scaffolding; change them only when adding/renaming the grammar, not for grammar logic.
